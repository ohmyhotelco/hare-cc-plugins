# Backend Implementation Plan

Analyze a functional specification (planning-plugin output) and produce a structured backend implementation plan (`plan.json`) that drives spec-aware scaffold generation and TDD implementation.

## Instructions

### Step 0: Validate Configuration

1. Read `.claude/backend-springboot-plugin.json`
2. If missing, tell the user to run `/backend-springboot-plugin:be-init` first and stop
3. Read `config.architecture` — currently only `cqrs` is supported

### Step 0.5: Detect Mode

**Auto-detection** — check if `docs/specs/{feature}/.progress/{feature}.json` exists:

- If not found:
  > "No planning-plugin specification found for '{feature}'."
  > "Options:"
  > "1. Create specification first: `/planning-plugin:spec {feature}`"
  > "2. Use manual mode: `/backend-springboot-plugin:be-crud {EntityName} field:Type ...`"
  - Stop here.
- If found: proceed to Step 1

### Step 1: Validate Spec

1. Read `docs/specs/{feature}/.progress/{feature}.json`

2. Check `status`:
   - `"reviewing"` or `"finalized"` → proceed
   - If status is `"reviewing"`:
     > "The specification is in 'reviewing' status — unresolved review issues may exist."
     > "For the most accurate plan, finalize first: `/planning-plugin:review {feature}`"
     > "Continue with current spec?"
     - If the user declines, stop here.
   - `"drafting"` or `"analyzing"` →
     > "The functional specification is not yet complete (status: {status})."
     > "Please finish writing the specification, then try again."
     - Stop here.

3. Read two language settings:
   - `specLanguage`: extract `workingLanguage` from the spec progress file (for locating spec markdown files in `docs/specs/{feature}/{specLanguage}/`)
   - `workingLanguage`: read from the backend plugin config (`.claude/backend-springboot-plugin.json`) for user-facing communication. This ensures consistent output language across all backend skills.
4. Language name mapping: `en` = English, `ko` = Korean, `vi` = Vietnamese

**Communication language**: All user-facing output in this skill must be in {workingLanguage} (from backend config).

### Step 2: Check UI DSL

1. Check if `docs/specs/{feature}/ui-dsl/manifest.json` exists
   - exists → `uiDslAvailable = true`
   - not exists → `uiDslAvailable = false`

2. If `uiDslAvailable` is false, display recommendation:
   > "UI DSL is not available. The plan will be generated by inferring from the spec markdown."
   > "For more accurate entity/field definitions, running `/planning-plugin:design {feature}` is recommended."

### Step 2.5: Check Existing Plan

1. Check if `docs/specs/{feature}/.implementation/backend/plan.json` exists → `existingPlanExists`
2. If `existingPlanExists`:
   > "An existing backend plan was found (generated at {planGeneratedAt})."
   > "Options:"
   > "1. Regenerate — create a new plan from the current spec (overwrites existing plan)"
   > "2. Cancel — keep the existing plan"
   - If user chooses 2: stop here.
   - If user chooses 1: proceed (overwrite mode)
3. If not `existingPlanExists`: proceed

### Step 3: Launch Backend Planner Agent

Create the output directory if it doesn't exist:

```
docs/specs/{feature}/.implementation/backend/
```

**Subagent Isolation**: Pass only the specified parameters below. Do not include conversation history or user feedback from prior steps.

Launch the `backend-planner` agent with:

```
Agent(subagent_type: "backend-planner", prompt: "
  Analyze the functional specification for '{feature}' and produce a backend implementation plan.

  Parameters:
  - feature: {feature}
  - specDir: docs/specs/{feature}/{specLanguage}/
  - uiDslDir: docs/specs/{feature}/ui-dsl/ (available: {uiDslAvailable})
  - config: {serialized backend plugin config}
  - projectRoot: {cwd}
  - outputFile: docs/specs/{feature}/.implementation/backend/plan.json

  Follow the process defined in agents/backend-planner.md.
  Write the implementation plan to the outputFile path.
")
```

### Step 4: Display Summary

1. Read the generated `docs/specs/{feature}/.implementation/backend/plan.json`
2. Display the summary in the working language:

```
Backend Implementation Plan for '{feature}':

  Source: docs/specs/{feature}/ (status: {specStatus}, UI DSL: {available/not available})
  Architecture: {config.architecture}

  Entities:       {count} ({entity names})
  Commands:       {count} ({command names})
  Queries:        {count} ({query names})
  API Endpoints:  {count}
  Exceptions:     {count}
  Test Scenarios: {count}

  Entity scaffold order: {entityDependencyOrder}
  Skipped screens: {skippedScreens count} (frontend-only)

  Plan saved to: docs/specs/{feature}/.implementation/backend/plan.json

  Next steps:
    1. Review the plan and edit if needed
    2. Run /backend-springboot-plugin:be-crud {EntityName} to scaffold from plan
       or  /backend-springboot-plugin:be-crud --all {feature} to scaffold all entities
```

### Step 5: Update Spec Progress

1. Read `docs/specs/{feature}/.progress/{feature}.json`
2. Add or update the `implementation.backend` field:

```json
{
  "implementation": {
    "backend": {
      "status": "planned",
      "planFile": "docs/specs/{feature}/.implementation/backend/plan.json",
      "planGeneratedAt": "{ISO 8601}"
    }
  }
}
```

3. Write the updated progress file back. **Merge rule**: preserve all existing fields in the progress file — only add or update the `implementation.backend` fields shown above. Do not modify `implementation.frontend` or any other existing fields.

### Step 5.5: Ensure Progress Directory

Ensure the backend pipeline progress directory exists so that `be-crud` can write entity-level progress files:

1. Read `workDocDir` from plugin config
2. Create `{workDocDir}/.progress/` directory if it does not exist

Note: Individual entity progress files are created by `be-crud` (one per entity). `be-plan` does not create feature-level progress files to avoid orphan entries that no subsequent skill updates.
