# Homepage Plugin

A Claude Code plugin that applies tech stack and coding conventions for building company marketing and homepage websites with Astro.

## Tech Stack

### Runtime & Build
- Node.js 22.x LTS (>= 22.12)
- Package Manager: pnpm
- Build: Astro 5.x (SSG default, optional SSR via adapters)
- Language: TypeScript (strict)

### Core Framework
- Astro 5 (App-like static site generator with islands architecture)
- Zero JS by default — only hydrate interactive components
- File-based routing in `src/pages/`
- ViewTransitions for SPA-like page navigation

### UI Layer
- Tailwind CSS (`@astrojs/tailwind`)
- shadcn/ui (Radix-based, code owned by the project) — may be replaced with in-house design system
- Icons: Lucide (`lucide-react`) — may be replaced with in-house icon set
- React 19 via `@astrojs/react` — used only for interactive islands

### Content
- Astro Content Collections with Zod schemas for type-safe content
- MDX via `@astrojs/mdx` — blog posts, case studies, long-form content
- Optional: Headless CMS client (Sanity, Contentful) — configured via `contentStrategy`

### Internationalization (i18n)
- Astro built-in i18n routing or astro-i18n-aut
- Languages: configurable via `i18nLocales` (default: ko / en)
- Translation files in `src/i18n/{locale}.json`
- Date/time: `Intl.DateTimeFormat` / `Intl.RelativeTimeFormat`

### SEO
- Static HTML output — inherently SEO-friendly (no client-side rendering issues)
- `@astrojs/sitemap` for automatic sitemap generation
- JSON-LD structured data per page type (Organization, WebSite, Article, BreadcrumbList, FAQ, Product)
- Open Graph + Twitter Card meta tags per page
- `robots.txt` configuration
- Canonical URL handling
- `astro:assets` `<Image />` with Sharp optimization (width, height, alt required)

### Testing
- Vitest + @testing-library/react — interactive island component tests only
- Playwright — E2E and visual regression
- Lighthouse CI — performance budgets (target: 90+ on all categories)
- axe-core — accessibility auditing (WCAG 2.1 AA)

### Linting
- ESLint v9 flat config with `eslint-plugin-astro`, `typescript-eslint`, `eslint-plugin-react-hooks`
- `eslintTemplate` flag in `.claude/homepage-plugin.json` controls auto-generation
- Config auto-generated from `templates/eslint-config.md` when no ESLint config exists

### Deploy
- Vercel / Netlify / CloudFlare Pages (static adapter)
- CDN edge caching for global delivery
- Preview deploys per PR recommended

## What is NOT Used (vs frontend-react-plugin)

- No Zustand — static pages need no client state management
- No Axios — use Astro fetch at build time or CMS SDK
- No MSW — content is static/MDX, no mock-first development
- No CRUD patterns — no entity management, no stores
- No auth/RBAC — public marketing pages
- No TDD pipeline — audit-focused testing instead

## Conventions

- 2-space indentation
- `.astro` files for static sections, React `.tsx` for interactive islands only
- All user-facing text uses i18n translation functions
- `<Image />` from `astro:assets` for all images (never raw `<img>`)
- Responsive design: mobile-first, Tailwind breakpoints `sm:`, `md:`, `lg:`, `xl:`
- Semantic HTML: proper heading hierarchy (single `<h1>` per page, logical `<h2>`-`<h6>`)
- Accessibility: `aria-label` on icon-only buttons, `alt` on all images, proper focus management
- Variable-length text: apply `truncate` / `line-clamp-*`

### Astro Component Conventions
- `.astro` components are Server Components — rendered to static HTML at build time
- React islands use `client:` directives for hydration:
  - `client:load` — hydrate immediately (forms, mobile navigation)
  - `client:visible` — hydrate when visible in viewport (carousels, accordions)
  - `client:idle` — hydrate when browser is idle (analytics, non-critical UI)
- Section components are self-contained with typed props (Astro `Props` interface)
- Pages use Astro layouts via frontmatter `layout` property

### Content Collections
- Schema defined in `src/content/config.ts` using Zod
- Blog posts as `.mdx` files in `src/content/blog/`
- Frontmatter: title, description, publishedAt, author, tags, ogImage
- Query via `getCollection()` and `getEntry()` — type-safe at build time

### SEO Conventions
- Every page must define `<title>`, `<meta name="description">`, OG tags
- JSON-LD structured data in `<script type="application/ld+json">` per page
- Sitemap auto-generated by `@astrojs/sitemap`
- Canonical URL via `<link rel="canonical">`
- Images must have explicit `width`, `height`, `alt` attributes

## Architecture

- **Agents**:
  - `page-planner` — user input analysis → page plan (page-plan.json)
  - `section-generator` — .astro sections + React islands generation
  - `page-assembler` — section assembly → full pages + SEO metadata + i18n
  - `seo-reviewer` — SEO compliance review (6 dimensions)
  - `quality-reviewer` — code quality + accessibility review (6 dimensions)
  - `review-fixer` — direct fix for review issues
- **Skills**: `/homepage-plugin:hp-init`, `/homepage-plugin:hp-plan`, `/homepage-plugin:hp-gen`, `/homepage-plugin:hp-verify`, `/homepage-plugin:hp-review`, `/homepage-plugin:hp-fix`
- **External Skills**: `web-design-guidelines` + `vercel-composition-patterns` (from `vercel-labs/agent-skills`) — installed by hp-init
- **Configuration**: `.claude/homepage-plugin.json` (created by `/homepage-plugin:hp-init`)
- **Templates**: `section-catalog.md` (15 section patterns), `page-module.md` (page assembly), `seo-checklist.md` (SEO requirements), `eslint-config.md` (ESLint v9 config), `astro-conventions.md` (Astro 5 conventions)

### Communication Language
- Skills read `defaultLocale` from `.claude/homepage-plugin.json`
- All user-facing output (summaries, questions, feedback, next-step guidance) must be in the configured locale language
- Language name mapping: `en` = English, `ko` = Korean, `vi` = Vietnamese

## Page Generation Pipeline

```
hp-init → hp-plan → hp-gen → hp-verify → hp-review → hp-fix
                                                       ↓
                                                  hp-review (re-review)
```

### hp-plan — Interactive Page/Section Definition
- Pages and sections are NOT predefined
- User describes site purpose, pages, and per-page content in natural language
- AI matches descriptions to section catalog (15 canonical types) and proposes composition
- User confirms, modifies, or adds custom sections
- Optional Figma screenshot/URL analysis for design reference
- Output: `page-plan.json` per page + `layout-plan.json` for shared layout

### hp-gen — 3-Phase Code Generation
1. **Infrastructure**: Layout + shared components + i18n + SEO utilities + Content Collection config
2. **Sections & Pages**: section-generator → page-assembler per page (sequential)
3. **Verification**: tsc + astro build + ESLint

### hp-verify — Quality Verification
1. TypeScript check (`npx tsc --noEmit`)
2. ESLint (template fallback)
3. Astro build (`npx astro build`)
4. Lighthouse CI (performance/accessibility/SEO >= 90)
5. Component tests (interactive islands only)

### hp-review — 2-Stage Code Review
1. **SEO Review** (seo-reviewer): 6 dimensions — metadata, structured data, heading hierarchy, image optimization, sitemap/robots, performance indicators
2. **Quality Review** (quality-reviewer, only if SEO passes): 6 dimensions — accessibility WCAG AA, responsive design, component composition, TypeScript strictness, i18n completeness, Astro conventions

### hp-fix — Direct Fix
- No TDD classification — homepage sections are presentational
- Direct fixes for all review issues
- Maximum 3 rounds per issue, then escalate
- Verification after each fix batch

## Verification Philosophy

Same principle as frontend-react-plugin: **"Evidence before claims, always"**

5-Step Gate:
1. IDENTIFY — identify the target to verify
2. RUN — execute verification tools (tsc, astro build, lighthouse, etc.)
3. READ — review the full output (exit code, error count)
4. VERIFY — determine whether the output matches the claim
5. CLAIM — report the result citing evidence

Red Flags:
- "should work" / "probably fine" — do not use without running tools
- Do not substitute a previous run's result for the current verification
- Do not skip verification because "the change is small"

### TypeScript Check
- Astro projects: `npx tsc --noEmit` (Astro does not use composite tsconfig by default)
- If `tsconfig.json` contains `references` array: use `npx tsc -b 2>&1` instead

### Build Check
- `npx astro build` — full static site generation
- Check for zero errors in output

## State Files

```
docs/pages/{page-name}/
├── page-plan.json                    ← Page plan (sections, metadata, i18n)
├── .progress/{page-name}.json        ← Pipeline progress
└── .implementation/homepage/
    ├── generation-state.json         ← Generation progress per phase
    ├── review-report.json            ← Review results
    ├── fix-report.json               ← Fix results
    └── .lock                         ← Concurrent execution prevention

docs/pages/_shared/
└── layout-plan.json                  ← Shared layout definition
```

### State File Safety

State files are critical for pipeline continuity.

**Read-Modify-Write rule**: When updating state JSON files:
1. Read the **latest** file content immediately before writing
2. Merge only the fields being changed — preserve all existing fields
3. Write the complete merged object

**Lock file**: Skills that modify state files must acquire `docs/pages/{page-name}/.implementation/homepage/.lock` before starting work. Release on completion or failure. Stale locks (older than 30 minutes) are automatically removed.

### Progress State Machine

```
planned → generated → verified → reviewed → done
             ↓    \       ↓         ↓
        gen-failed  \ verify-failed review-failed
                     \              ↓
                      → fixing → (re-review → reviewed/review-failed)
                        escalated
```

Additional transitions:
- `generated → reviewed | review-failed | done` — hp-verify is optional
- `verify-failed → reviewed | review-failed | done` — hp-review accepts verify-failed
- `verified | verify-failed → fixing` — hp-fix accepts verification results directly (without hp-review)
- `gen-failed → generated | gen-failed` — re-run hp-gen
- `fixing → reviewed | review-failed` — after hp-fix, hp-review determines next status
- `escalated → fixing | verified | reviewed` — after manual intervention, re-enter pipeline via hp-fix, hp-verify, or hp-review

## Project-Level Configuration

`.claude/homepage-plugin.json` (created by `/homepage-plugin:hp-init`):
```json
{
  "framework": "astro",
  "contentStrategy": "mdx",
  "i18nLocales": ["ko", "en"],
  "defaultLocale": "ko",
  "deployTarget": "aws",
  "eslintTemplate": true
}
```

- `framework`: `"astro"` — reserved for future expansion
- `contentStrategy`: `"mdx"` (default) | `"headless-cms"` | `"both"`
- `i18nLocales`: array of locale codes (default: `["ko", "en"]`)
- `defaultLocale`: default locale code (default: `"ko"`)
- `deployTarget`: `"aws"` (default) | `"vercel"` | `"netlify"` | `"cloudflare"` | `"static"` — `"aws"` uses S3 + CloudFront static hosting
- `eslintTemplate`: `true` (default) | `false` — auto-generate ESLint config
